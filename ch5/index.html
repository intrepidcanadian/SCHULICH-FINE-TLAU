§5.2 · The double-spend problem · SHA-256
04 / 17
Hashing · cryptographic foundation

Solving the double-spend problem.

Earlier digital monies failed because no one could verify a coin had not already been spent — without a central mint. Satoshi’s solution: a time-stamped distributed ledger secured by cryptographic hashing.

Blockchain
Distributed public ledger
Transactions batched into 1 MB blocks every ~10 minutes. Each block: timestamp, prev-block hash, transactions, nonce. Append-only, immutable, transparent.
Hashing · SHA-256
One-way fingerprint
Maps data of any size to a 256-bit string. Identical input ⇒ identical hash. Cannot be inverted with current computing power. Quantum computing threatens this.
Pseudo-anonymity
Public + private keys
Owner identified by hashed public key (digital address). Private key unlocks transfers. Real-life identity hidden but traceable.
Five Building Blocks of Bitcoin
01 · Blockchain
Distributed public ledger. Pages of transactions chained by hashes.
02 · Hashing
SHA-256 secures, compresses & chains blocks.
03 · P2P
46,000+ nodes (2023) running Bitcoin Core software.
04 · Keys
Public & private. Cryptographic proof of ownership.
05 · Script
Programming language to execute send / receive actions.
· Consensus
Proof-of-work mining randomly selects who appends the next block.
Throughput: ~1,500 transactions per 10-min block = 2.5 TPS · vs. VISA up to 24,000 TPS. Block size capped at 1 MB.

§5.2 · Technical · block header
04b / 17
Technical deep dive · what is actually hashed

A block is 80 bytes of header · plus transactions.

Miners do not hash the whole block. They hash the 80-byte block header. The transactions are committed to via a single 32-byte Merkle root that lives inside the header — a clever compression trick that lets light clients verify membership without downloading every transaction.

Block header · 80 bytes
04 B   version     consensus rules in effect
32 B   prev_block_hash   chains to parent (the “chain”)
32 B   merkle_root    SHA-256(SHA-256(tx tree))
04 B   timestamp     Unix seconds, ±2 hr tolerance
04 B   bits          compact-encoded difficulty target
04 B   nonce         32-bit search space (miners cycle this)
When the 32-bit nonce range is exhausted, miners increment the coinbase extraNonce, which changes the Merkle root, giving them effectively unlimited search space.
Block hash = SHA-256(SHA-256(80-byte header)). Bitcoin uses double SHA-256 to defend against length-extension attacks — legacy hardening from the SHA-2 era.
— Bitcoin Core source · Nakamoto (2008)
Merkle tree · hash the tree, not the trades
ROOT / \ H(AB) H(CD) / \ / \ H(A) H(B) H(C) H(D) | | | | tx_A tx_B tx_C tx_D
A Merkle proof for tx_C is just {H(D), H(AB)} — log₂(n) hashes. A light client (SPV wallet) verifies inclusion without storing the chain.

§5.2 · Technical · SHA-256 properties
04c / 17
Technical deep dive · the math that makes mining hard

One bit flips. The whole hash explodes.

A cryptographic hash function must satisfy three properties: pre-image resistance (can’t reverse), second pre-image resistance (can’t collide deliberately), and the avalanche effect (1-bit input change ⇒ ~50% of output bits flip). Miners exploit the avalanche effect by iterating the nonce until a hash falls below the target.

Avalanche effect · worked example
input: “hello”
SHA-256: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
input: “Hello”  (only the “h” case flipped)
SHA-256: 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
Hamming distance ≈ 50% of bits — no pattern, no shortcut.
One-wayness: given a 256-bit output, there is no algorithm faster than brute force — on average 2²⁵⁵ attempts — to find any input that produces it. Quantum (Grover) reduces this to 2¹²⁸, still astronomical.
— NIST FIPS 180-4 · SHA-2 family
SHA-256 pipeline · one block at a time
message → pad to 512-bit chunks ↓ [8 × 32-bit state words] ↓ (×64 rounds per block) Maj / Ch / Σ₀ / Σ₁ + K_t + W_t ↓ compress → next chunk ↓ final state = 256-bit digest
Each compression round mixes bits via shifts, XORs, and modular addition. Bitcoin ASICs implement this directly in silicon, achieving ~100 TH/s per chip.
Output
256-bit
Always 32 bytes · 64 hex chars
Network hash rate
500+ EH/s
5 × 10²⁰ SHA-256 ops / second (2024)

§5.2 · Proof-of-work, mining & PoS
05 / 17
Consensus mechanism · PoW

Mining: random number guessing.

Miners race to find a nonce that — when hashed with block contents — produces a hash with N leading zeros. The first to find it earns the block reward. Difficulty auto-adjusts to keep blocks 10 min apart.

§5.2 · Technical · the mining loop
05b / 17
Technical deep dive · what the ASIC is actually doing

Mining = brute-forcing one inequality.

A miner builds a candidate block, then loops the nonce until the 256-bit double-SHA-256 of the header is numerically less than the target. The target is encoded in the 4-byte bits field. As the network hash rate grows, the difficulty bomb shrinks the target, making valid hashes rarer.

Mining loop · pseudocode
def mine(header, target): for nonce in range(0, 2**32): header.nonce = nonce h = sha256(sha256(serialize(header))) if int(h, 16) < target: return nonce, h # ← valid block! # 4B nonce exhausted — # bump extraNonce in coinbase, rebuild Merkle root, retry
No structure. No shortcut. Just a tight hash & compare loop. ASICs do this in dedicated silicon at ~100 TH/s per chip; a modern S21 box runs ~200 TH/s.
Probability of success per hash ≈ target / 2²⁵⁶. At a difficulty of D, expected hashes to find a block ≈ D × 2³². A miner with hash rate h wins block reward at rate h / Hnetwork.
— Bitcoin Core · chainparams.cpp
Difficulty adjustment · every 2,016 blocks
new_target = old_target × ( actual_time(2016 blocks) / expected_time = 2016 × 10 min ) clamp factor ∈ [1/4 , 4] expected window = 2 weeks
If the last 2,016 blocks were mined faster than 2 weeks, target shrinks (harder). If slower, target grows (easier). The clamp prevents oscillation. Keeps block-time at ~10 min regardless of total hash power.

§5.2 · Technical · UTXO & scripts
05c / 17
Technical deep dive · what a Bitcoin payment really is

Bitcoin has no accounts. Only unspent outputs.

The Bitcoin state is not a list of balances — it is a set of Unspent Transaction Outputs (UTXOs), each locked to a scriptPubKey. To spend, supply a scriptSig that satisfies the lock. A transaction consumes UTXOs (inputs), creates new ones (outputs); the rest is fee.

Transaction structure
tx { version: 2 inputs: [ { prev_tx: 0xabc..., vout: 1, scriptSig: <sig> <pubKey> } ] outputs: [ { value: 0.40 BTC, // to Bob scriptPubKey: P2PKH(hash160(pubKey_B)) }, { value: 0.09 BTC, // change to self scriptPubKey: P2PKH(hash160(pubKey_A)) } ] // fee = sum(in) − sum(out) = 0.01 BTC }
Inputs must be consumed entirely — if Alice has a 0.50 BTC UTXO and wants to pay 0.40, she must create a 0.09 change output back to herself (the rest is fee).
P2PKH script · the most common lock
# scriptPubKey (lock, in the output) OP_DUP OP_HASH160 <pubKeyHash> OP_EQUALVERIFY OP_CHECKSIG # scriptSig (key, supplied by spender) <signature> <pubKey> # Stack execution: push sig, push pubKey, DUP, HASH160, compare to pubKeyHash, CHECKSIG → TRUE/FALSE
Bitcoin Script is a Forth-like, non-Turing-complete stack language — no loops, no recursion. Deliberately limited to keep validation deterministic and cheap.
Why UTXO? Each output can be validated independently — perfect for parallel verification and pruning. The cost: wallets must track many small “coins”, and dust outputs bloat the UTXO set (~5 GB and growing in 2024).

§5.5 · Wallets, exchanges, honey pots
08 / 17
Storage and trading — the two pain points

Paper, software, hardware — and exchanges as honey pots.

A cryptowallet stores private keys, not coins. Hot wallets (software) connect to the internet; cold wallets (hardware) keep keys offline. Trezor (2014) was the first hardware wallet. Cryptoexchanges — Binance, Coinbase, Kraken — trade 24/7 across 400+ unregulated venues.

Wallet evolution
01 · Paper
Print keys, lock them away. Lost, stolen, destroyed.
02 · Software
Hot wallet. Desktop, tablet, smartphone. Free, exposed.
03 · Hardware
Cold wallet. Trezor (2014), Ledger. PIN + passphrase, USB.
Key components
Address: public identifier for receiving payments. Private key: secret number for signing transfers. Recovery seed: word list to restore a lost wallet.
When you deposit coins on an exchange, the exchange holds the private keys — they own the coins on-chain. The exchange becomes a honey pot.

§5.7 · CBDC & risks to HODLers

Money: unit of account, medium of exchange, store of value

CBDCs & the six risks to HODLers.

Central banks are responding with digital fiat. The BIS reports 80+ countries are exploring CBDCs. Most focus on wholesale (interbank), not retail. Crypto enthusiasts are unimpressed.

Critique of Bitcoin as money
Few merchants accept BTC for payment. Mining takes hours-to-days to settle. Extreme volatility undermines its store-of-value function.
Three open questions on CBDCs
1. Blockchain or centralized ledger? · 2. Does CBDC threaten monetary policy and stability? · 3. Retail or wholesale only?
A CBDC controlled by a central bank does not require a blockchain — there’s a trusted party. But a truly decentralized, P2P CBDC would need one.

§5.8 · Cryptoeconomics research
11 / 17
Three bodies of academic research

Trust, throughput, and the speed of light.

Cryptoeconomics applies theory and empirics to Bitcoin and blockchain. Three findings dominate: trust is broken, the network cannot scale, and high volatility is structural — not transient.

01 · Trust
Illegal activity & manipulation
One study: 25% of users, 50% of transactions 2009–2017 tied to illegal activity. Disputed by follow-ups (<3% volume). Suspicious addresses move BTC price. Stigma lingers.
02 · Operational efficiency
Bitcoin cannot scale
Block size 1 MB limits TPS. Miners’ concentrated hash power = natural oligopoly. As network expands, consensus delays grow. Adoption is an equilibrium ceiling.
03 · Stability
Volatility is a feature
Multiple equilibria with different price/security levels. Price feedback amplifies booms and busts. Sunspot-like volatility unrelated to fundamentals.
Some researchers conclude Bitcoin is an intrinsically worthless, non-dividend-paying object that will never replace the US dollar — while blockchain itself may still be transformational.
— Cryptoeconomics literature, 2018–2023
Why miners may collude
Pools controlling ~75% of hash power may strategically leave blocks partially empty to extract higher fees from urgent users, crowding out small payments.
Six discussion questions for crypto
Can BTC be a global currency? What are actors’ incentives? What governance resolves moral hazard? Asymmetry between miners/users? Best consensus algo? Drivers of returns?
